- [x] [Code, 2026-05-01] **Watcher reconnect on realtime drop (instead of fail-fast)**: a Supabase realtime hiccup on 2026-05-01 06:23 local triggered the v0.9.0 fail-fast logic, exiting the process. With no supervisor configured (the watcher runs as a detached `node.exe` per `watcher_runtime` memory), the watcher silently stayed dead until manual restart, leaving a real morning request stuck in `pending`. Fix: on `CHANNEL_ERROR` / `TIMED_OUT` / `CLOSED`, reconnect in-process with exponential backoff (2s → 4s → 8s → 16s → 32s → cap 60s). On successful re-`SUBSCRIBED`, sweep that channel's table for any pending rows that landed during the dead window (realtime doesn't replay missed INSERTs). Shutdown handler now cancels pending reconnect timers cleanly. `uncaughtException` / `unhandledRejection` keep their fail-fast behavior (those reflect bugs, not transient network issues).
