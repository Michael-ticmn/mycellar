cellar27how a request travels

Your phone never talks to your laptop. Both talk to Supabase. The laptop watcher does the AI reasoning and writes the answer back. Your phone sees it appear.

1
Phone inserts a pairing_request with a snapshot of your cellar and the dish. Anon key + RLS confines it to your user_id.
2
Supabase Realtime publishes the new row over a websocket to every subscribed client.
3
Laptop watcher hears the event, runs the policy gate (allowlist + rate limit), atomically claims the row (pending → picked_up), renders a markdown request file in ~/cellar27-bridge/requests/, and spawns claude --print at it. Claude reads the file, reasons, writes the response markdown to responses/.
4
Watcher notices the response file (chokidar), parses it, inserts a pairing_response row, marks the request completed, archives both files into processed/.
5
Realtime pushes the new response row to the phone. Frontend renders bottle cards + narrative.
Liveness
  • Laptop only awake during step 3 (~20–60s)
  • Phone offline mid-flight: catches up on reconnect
  • Laptop asleep on submit: row queues; watcher's startup sweep picks it up when laptop wakes
Lockdown
  • Sign-ups disabled in Supabase
  • service_role key only in watcher/.env
  • RLS + allowlist + per-user rate limit
  • DB CHECK constraints + pending-request cap
  • HTTPS only · PWA-installable
